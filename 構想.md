# サーバーレスP2P通信によるトランプゲーム開発の検討

## 1. 背景・目的

サーバーを介さずにユーザー間で直接通信し、トランプゲーム（ババ抜き）を実現する方法を検討した。サーバー運用コストをゼロに抑えることを目標とする。

## 2. P2P通信の基本

P2P（Peer-to-Peer）通信とは、サーバーを中継せずにユーザー同士が直接データをやり取りする通信方式である。代表的な技術として以下がある。

- **WebRTC** — ブラウザ間で音声・映像・データを直接通信できる。最も広く使われるP2P技術
- **BitTorrent** — ファイル共有に特化した分散型プロトコル
- **IPFS** — 分散型のファイルシステム

ただし、いずれの方式でも接続確立時にはシグナリング（接続情報の交換）が必要であり、完全にサーバーゼロでの運用には工夫が求められる。

## 3. トランプゲームとP2Pの相性

トランプゲーム（特にババ抜き）はP2P通信と非常に相性が良い。

- 通信データが極めて軽量（カードの操作情報のみ、数十バイト程度のJSON）
- リアルタイム性の要求が低い（ターン制）
- 同時接続数が少人数

通信データの例:

```json
{"action": "draw", "index": 3}
```

### 設計上の課題

P2Pではサーバーという中立な管理者が存在しないため、カードのシャッフルや配札の公平性をどう保証するかが課題となる。対策としては以下がある。

- **Mental Poker Protocol** — 暗号的手法で公平性を保証する方式
- **ホスト方式** — 片方のプレイヤーがホスト役としてゲームロジックを管理する方式（カジュアル用途向き）

## 4. 実現方法の比較

### 4.1 C++ + TCPソケット通信（LAN向け）

- BSDソケット / Winsockで直接TCP接続
- 片方がホスト（サーバー役）、もう片方がクライアント
- LAN内ならIPアドレス直接入力で接続可能
- まずLAN対戦で動作確認し、その後インターネット越しに拡張するステップが推奨

### 4.2 Bluetooth通信（スマートフォン向け）

- サーバーが一切不要で完全ゼロ円を実現可能
- Android: Bluetooth Classic / BLE のAPIが利用可能
- iOS: Multipeer Connectivity フレームワーク（BLE + Wi-Fi Direct を自動管理）
- 注意: Bluetoothの制御はOS固有APIに依存するため、C++でゲームロジック、ネイティブ言語（Java/Kotlin, Swift）でBluetooth層を実装する構成が現実的

### 4.3 WebRTC（ブラウザ向け）

ブラウザ間のP2P通信にはWebRTCが最適。なお、Web Bluetooth APIはBLEデバイス（IoT機器等）向けであり、スマホ同士のP2P通信には使用できない。

シグナリング（接続情報の交換）方法によってコストが変わる。

| シグナリング方式 | サーバーコスト | 実用性 |
|---|---|---|
| Firebase無料枠 / Cloudflare Workers無料枠 | ほぼゼロ | 高い |
| QRコードで接続情報を交換 | 完全ゼロ | やや手間 |
| チャットツールでSDP文字列を交換 | 完全ゼロ | 実用的 |

## 5. チャットツールによるシグナリング（推奨案）

サーバー完全ゼロ円かつ実用的な方法として、WebRTCの接続情報（SDP）をチャットツール経由で交換する方式が有力である。

### 接続フロー

1. **ホスト側** がWebRTCの「Offer SDP」を生成し、テキストとしてコピー
2. ホストがLINE・Discord等のチャットツールで相手にOffer SDPを送信
3. **ゲスト側** がOffer SDPを貼り付け、「Answer SDP」を生成
4. ゲストがAnswer SDPをチャットツールでホストに返信
5. **ホスト側** がAnswer SDPを貼り付けると、P2P接続が確立
6. 以降はサーバーなしで直接通信が行われる

### SDPの特徴

- 数百〜数千文字程度のBase64風テキスト
- チャットに貼り付けて交換可能な範囲
- 圧縮やエンコードによる短縮も可能

## 6. まとめ

| 項目 | 結論 |
|---|---|
| 最適なゲーム | ババ抜き（軽量データ、ターン制、少人数） |
| 最適なP2P技術 | WebRTC DataChannel（ブラウザ向け） |
| サーバーコスト最小化 | チャットツールでSDP交換 → 完全ゼロ円 |
| 公平性の担保 | ホスト方式（カジュアル用途では十分） |
| 開発ステップ | ブラウザ版プロトタイプ → 機能拡張の順が推奨 |

サーバーを一切使わずにブラウザ上でトランプゲームを実現することは十分に可能であり、チャットツールをシグナリング手段として活用することで、実用性とコストゼロを両立できる。
